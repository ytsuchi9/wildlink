# WES 2026 プロトコル・リファレンス
(WildLink Event Standard 2026 Protocol Reference)

## 第1章：アーキテクチャと通信設計思想

### 1.1 グループ概念と完全疎結合 (Configuration & Decoupling)
WES 2026では、ハードウェア、ロジック、UIの各レイヤーを環境変数(`.env`)とマニフェストによって完全に分離し、「完全疎結合」な分散システムを実現する。

* **GROUP_ID**: 複数のHubやNodeを論理的な1つのシステムとして束ねるID（例：`site_alpha`）。マルチテナントや複数サイトの混線を防ぐ。
* **SYS_ID**: 各物理デバイス（Raspberry Pi等）のユニークID（例：`node_001`）。
* **VST_ROLE_NAME**: 物理的な接続インターフェース（USB等）を隠蔽する論理的な役割名（例：`cam_main`, `cam_sub`）。
* **設定のポータビリティ**: 接続先DB、MQTTブローカーIP、ポート番号などはすべて `.env` およびDB (`node_configs`) で管理し、コード内へのハードコードを禁止する。

### 1.2 MQTTルーティングとトピック構造
非同期イベントとUIキックの到達性を担保するため、以下の階層構造でメッセージをルーティングする。

* **基本フォーマット**: `wildlink/{group_id}/{sys_id}/{role}/{type}`
* **トピック種別**:
  * `.../cmd`: 命令の受信（UI/Hub → Node）
  * `.../event`: 状態変化の通知、UIのキック（Node/Hub → UI）
  * `.../log`: システム全体の診断・エラーログ（全モジュール → Logger）

### 1.3 ハイブリッド通信モデル：ポーリング vs MQTTキック
システムの「一貫性」と「即応性」を両立するため、用途に応じて経路を使い分ける。

| 手法 | 経路 | 主な用途 | 特徴 |
| :--- | :--- | :--- | :--- |
| **ポーリング** | UI ↔ DB | ダッシュボードの全体同期、死活監視、過去のログ取得 | DBが「真実のソース(SSOT)」となる。再起動後も状態を完全復元可能。 |
| **MQTTキック** | Hub/Node → UI | 映像ウィンドウの自動ポップアップ、即時警告、コマンド応答 | ゼロ遅延。DBへのクエリ負荷をかけずにUIのDOM操作（画面遷移）をトリガーする。 |


## 第2章：WMP (WildLink Media Protocol) 仕様

### 2.1 パケット・ヘッダー構造 (32 Bytes / Big-Endian)
`common/wmp_core.py` にて定義される、軽量かつ高速なUDP送信用バイナリ構造。

| Offset | Size | Name | Type | Description |
| :--- | :--- | :--- | :--- | :--- |
| 0 | 4 | `MAGIC` | String | `b"WMP!"` (固定識別子) |
| 4 | 12 | `sys_id` | String | 送信元ノードID (Null-padded, 例: `node_001`) |
| 16 | 4 | `info` | UInt32 | `Media(8bit)` \| `Bus(8bit)` \| `Flags(16bit)` の結合データ |
| 20 | 4 | `p_len` | UInt32 | ペイロード(データ部)のバイト長 |
| 24 | 4 | `seq` | UInt32 | **フレームID** (フレームごとにインクリメント, 0-4294967295) |
| 28 | 2 | `f_idx` | UInt16 | チャンク番号 (現在のパケットのインデックス, 0開始) |
| 30 | 2 | `f_total` | UInt16 | 総チャンク数 (このフレームを構成するパケットの総数) |

### 2.2 Infoフラグ詳細 (16-bit Flags)
`info` 内の下位16ビット(`Flags`)は、ペイロードの特性を示すために使用する。これにより将来的な拡張（音声やセンサーデータの多重化）に対応する。

* `0x0001`: **Keyframe (I-Frame)** - 映像の基準フレーム。UI側でのデコード開始のトリガーとなる。
* `0x0002`: **Contains Audio** - 音声データがマルチプレックスされている場合。
* `0x0004`: **Telemetry / Sensor** - 映像ではなく、バイナリ化されたセンサーデータ等を含む場合。
* `0x0008`: **EOF (End Of Stream)** - 配信の明示的な終了フラグ。

### 2.3 アセンブリとメモリ管理 (Hub側)
1. **初期化**: 新しい `seq` (フレームID) を受信した際、`f_total` 分の配列(チャンクバッファ)をメモリ上に確保する。
2. **結合**: `f_idx` に従いペイロードを格納。到着順序が前後してもインデックスにより正しく再構築される。
3. **破棄とタイムアウト**: 新しい `seq` が到着した時点で、未完成の古い `seq` のバッファは「パケットロス」とみなし即座に破棄（メモリ解放）する。


## 第3章：イベント・シーケンスとUI連動

### 3.1 正常系：ストリーム開始シーケンス
UIのボタン押下から映像が表示されるまでの、無駄のないキックフロー。

1. **[UI]** ユーザーが配信開始ボタンを押下。
2. **[UI]** `node_commands` テーブルに `{"act_run": true}` をINSERT。（ポーリング待ちせず待機）
3. **[Node]** コマンドを検知し、プロセス（カメラ等）を起動。
4. **[Node]** 最初のフレーム生成に成功した時点で `val_status = "streaming"` をDBに同期。UDP送信開始。
5. **[Hub]** UDPで最初のパケットを受信し、映像を組み立てる。
6. **[Hub]** 初回受信をトリガーとして、MQTTで `stream_ready` イベントを送信。
7. **[UI]** MQTT経由で `stream_ready` を受信。即座に映像ウィンドウをポップアップさせ、MJPEGストリームの読み込みを開始する。

### 3.2 異常系：プロセス起動失敗シーケンス
ハードウェア未接続やドライバーエラー時の「正直な報告」フロー。

1. **[Node]** コマンドを受けプロセスを起動するが、直後にエラー終了（Exit Code > 0）。
2. **[Node]** 標準エラー出力をキャッチし、`val_status = "error"` をセット。
3. **[Node]** `send_response` により、`log_code = 500` と詳細なエラーメッセージをDBに記録。
4. **[Node]** MQTTで `stream_error` イベントを送信。
5. **[UI]** UIはキックを受け取り、画面上に警告バッジ（「カメラが見つかりません」等）を即座に表示する。

### 3.3 異常系：ネットワーク切断（タイムアウト）シーケンス
稼働中に物理的な切断が発生したフェイルセーフ。

1. **[Hub]** 配信中、5秒間該当ポートにUDPパケットが到達しないことを検知。
2. **[Hub]** 自動的にDBの状態を `idle` に戻し、`log_code = 408` (Timeout) を記録。
3. **[Hub]** MQTTで `stream_lost` イベントを送信。
4. **[UI]** キックを受け取ったUIは、ストリームウィンドウを自動的に閉じるか、「NO SIGNAL」画面に切り替える。


## 第4章：エラーコード体系とログ管理

### 4.1 HTTP × POSIX ハイブリッド・エラーモデル
システム制御の厳密さとWeb UIとの親和性を高めるため、DBおよびイベント通信上の `log_code` はHTTPステータスコードの概念を拡張して運用する。

### 4.2 イベント・ステータス一覧 (`val_status`)
デバイスの現在の物理状態を示す絶対値。

* `idle`: 待機中。リソースは完全に解放されている。
* `starting`: 起動または初期化プロセス実行中。
* `streaming`: 正常稼働・データ配信中。
* `paused`: 一時停止。プロセスは維持しつつ配信を抑制している状態。
* `error`: 異常終了または実行不可能な状態。

### 4.3 主要ログコード (`log_code`) リファレンス

| Code | Status | Category | Description |
| :--- | :--- | :--- | :--- |
| **200** | OK | Success | 正常終了、または正常に稼働中。 |
| **202** | Accepted | Success | 命令を受理し、非同期で処理を開始した。 |
| **400** | Bad Request | Request Error | `cmd_json` のパース失敗、無効なパラメータ。 |
| **404** | Not Found | Hardware Error | 指定されたデバイス（USBカメラ等）が物理的に見つからない。 |
| **408** | Request Timeout | Network Error | 一定時間パケットが到達せず、タイムアウトした。 |
| **409** | Conflict | Logic Error | 状態の矛盾（例: すでに実行中のデバイスに対する開始命令）。 |
| **500** | Internal Error | Process Error | 実行プロセスの異常終了、セグメンテーションフォールト。 |
| **503** | Service Unavail | Resource Error | デバイスがビジー状態、またはリソース（メモリ/CPU）不足。 |

### 4.4 拡張ログ (`log_ext`) の活用
静的なエラーメッセージに加え、`node_status_current` および `node_commands` の `log_ext` カラムにJSON形式で詳細な診断情報を格納する。
* **例**: `{"stderr": "libcamera: failed to map buffer", "exit_code": 255, "port": 5006}`
* これにより、管理画面のUI上でエラーの詳細をツールチップ等で展開することが可能となる。
