# WildLink 2026 指令系統・設計コンセプト

## 1. 概要
「Web ↔ Hub ↔ Node」間の指令を、単なる状態の同期（Sync）から、意図と結果を伴う「履歴（Audit Log）」へとアップグレードする。

## 2. node_commands テーブルの役割拡張
これまでの「リロード通知」に加え、具体的な変更内容（差分）を記録する。

- **cmd_json (変更パッチ)**: 
    - `{"action": "update", "target": "cam_main", "params": {"val_res": "1280x720"}}` のように、変更点そのものを封入。
- **log_code (ステータス記号)**: 
    - log_code 運用ルール:
        原則として HTTPステータスコード をベースにした3桁の数字を使用する。
        必要に応じて 404-NODEV のように、ハイフンで詳細情報を付与しても良い。
        これにより、Webフロントエンド側では code >= 400 であれば一律でアラート（赤色）を出す、といった汎用的な処理が可能になる。

- **log_msg (詳細ログ)**: 
    - Node側での実行詳細やエラーの生メッセージをJSONまたはテキストで記録。

## 3. WebUI（Rack Console）の挙動
- **ラック構造**: `device_catalog` の `ui_type` に基づき、ユニットを動的に配置。
- **双方向バインディング**: 
    1. ボタン押下で `node_commands` に `pending` 行を生成。
    2. Nodeからの返信（`log_code` / `val_status`）をJSが監視し、ボタンの状態をリアルタイム更新。

## 4. Node（MainManager）の挙動
- 受信した `cmd_json` の `action` に応じて、設定のマージ、削除、同期を柔軟に実行。
- 処理完了後、`log_code` カラムに結果を書き戻すことで、Web側へのフィードバックを完結させる。
