## 1. WES 2026 拡張規格：ログ管理ユニット (Log Management Unit)

### 概要
システム内における各ノードの動作ログ、エラーログ、および各種デバッグ情報を収集・配信するための仮想サービスユニット（VST）の仕様を定義する。ログ出力を一つのデバイスとして抽象化することで、他のセンサーと同様のMQTTフローで制御可能とする。

### ユニット定義
* **vst_type:** `sys_logger`
* **vst_role_name (推奨):** `log_sys`, `log_app`
* **主要な役割:** 指定されたパスのログ読み取り、フィルタリング、およびMQTTへの定時/即時配信。

### カタログ定義 (device_catalog):
* vst_type: sys_logger
* default_params: 動作モードやログパスのデフォルトJSONを格納。

### 個別設定 (node_configs):
* vst_role_name: MQTTトピックの第3階層（例: log_sys）として使用。
* val_params: 実際の動作設定（kick / polling の切り替え等）。

### 動作モード (val_params)
`val_params` 内の `log_mode` キーによって、配信挙動を動的に切り替える。

| モード | 説明 | メリット |
| :--- | :--- | :--- |
| **kick (Event-Driven)** | サーバーからの `fetch_log` 命令をトリガーに1回配信。 | 通信帯域とCPU負荷の最小化。 |
| **polling (Time-Driven)** | `val_interval` で指定された秒数ごとに最新分を自動配信。 | リアルタイムな監視が必要なフェーズに有効。 |

### MQTT トピック構造
* **Command:** `nodes/{node_id}/{role}/cmd`
    * `{"action": "fetch_log", "cmd_id": 123}` : ログの即時取得命令。
    * `{"action": "set_mode", "log_mode": "polling"}` : モードの動的変更。
* **Event:** `nodes/{node_id}/{role}/event`
    * `{"event": "log_fetched", "val_status": "success", "ref_cmd_id": 123}` : 取得完了通知。
* **Data (Env):** `nodes/{node_id}/{role}/env`
    * `{"env_log_msg": "...", "val_level": "INFO", "val_time": "ISO8601"}` : ログデータ本体。

## 2. 実装用 SQL スクリプト

DBに「ログ管理ユニット」という概念を登録し、実際のノード（例として `node_001`）に適用するためのSQLです。

### A. device_catalog への登録 (マスタ定義)
まずは「システムロガー」という型を定義します。

```sql
INSERT INTO device_catalog (
    vst_type, 
    vst_class,
    ui_component_type,
    default_params, 
    log_note
) VALUES (
    'sys_logger', 
    'system',
    'log_viewer',
    '{"log_mode": "kick", "val_interval": 300, "log_path": "/var/log/mqtt_agent.log", "max_lines": 50}', 
    'システムおよびアプリケーションのログ収集用仮想ユニット'
);
```

### B. node_configs への登録 (ノード別設定)
特定のノードに対して、ログユニットを有効化します。ここではデフォルトを「キックモード（節約モード）」に設定しています。

```sql
-- node_001 にログユニットを追加
INSERT INTO node_configs (
    sys_id, 
    vst_type, 
    vst_role_name, 
    val_enabled, 
    val_params, 
    vst_description
) VALUES (
    'node_001', 
    'sys_logger', 
    'log_sys', 
    1, 
    '{"log_mode": "kick", "val_interval": 300, "log_path": "/var/log/mqtt_agent.log", "max_lines": 20}', 
    'メインシステム・ログ収集ユニット'
);
```

--------------
ログ出力を「独立したひとつの仮想ユニット（VST）」として扱うことで、カメラやセンサーと全く同じ仕組み（MQTTトピックやコマンド処理）で制御できるようになります。

---

### 1. `device_catalog` への登録（マスタ定義）
まず、システム全体で「ログ管理ユニット」という機能が存在することを定義します。

* **vst_type:** `sys_logger` (または `unit_logger`)
* **vst_description:** 「システムログ・センサーログ収集ユニット」
* **初期設定値のテンプレート:** `val_params` にデフォルトのJSON構造を持たせておくと親切です。

### 2. `node_configs` への設定（インスタンス化）
特定のラズパイ（例：`node_001`）に、このログユニットを割り当てます。

* **vst_role_name:** `log_sys` や `log_main`（トピック名 `nodes/node_001/log_sys/...` に反映されます）
* **val_enabled:** `true` (有効) / `false` (無効)
* **val_params:** ここが今回の肝となる**「モード切替」**の格納場所です。

#### `val_params` の中身（JSON例）
```json
{
  "log_mode": "kick",     // "polling" または "kick"
  "val_interval": 300,    // ポーリング時の間隔（秒）
  "log_path": "/var/log/mqtt_agent.log", // 読み取る対象
  "act_priority": "high"  // ログの重要度
}
```

---

### 3. この設計にするメリット

この方法で進めることには、非常に大きなメリットが3つあります。

1.  **疎結合の維持:**
    Pythonエージェント側は「自分がどの役割（Role）を持っているか」を `node_configs` から読み取るだけで済みます。カメラ制御ロジックを汚さずにログ機能を「アドオン」として追加できます。
2.  **ブラウザUIとの親和性:**
    `sensor-unit.js` や `camera-unit.js` と並べて **`log-unit.js`** を作るだけで、管理画面に「ログ取得ボタン」や「現在のログ表示」を簡単に追加できます。
3.  **動的なモード変更:**
    ブラウザから `action: "set_params", "log_mode": "polling"` というコマンドを投げれば、DBを書き換えつつ、MQTT経由でエージェントの動作をリアルタイムで切り替えられます。

---

### 4. 実行イメージ（キックモードの場合）

1.  **ブラウザ:** `log-unit.js` の「ログ更新」ボタンを押す。
2.  **PHP:** `nodes/node_001/log_sys/cmd` へ `{"action": "fetch_log", "cmd_id": 999}` を発行。
3.  **Python:** 命令を受け取り、`val_params` のパスからログを読んで、`nodes/node_001/log_sys/env` にデータを流す。
4.  **ブラウザ:** ログが表示される。



### まとめ：理解の確認
「`device_catalog` に型を作り、`node_configs` で実体を作る」という理解で**完璧に合っています。** これにより、ログ出力さえも一つの「デバイス」として抽象化される、非常に美しいアーキテクチャになります。
